v14 — Suggested recipient rules

Issue #64 part 1 — the pure-heuristic suggestion surface. The app mines the last 60 days of transactions for repeat merchants / VPAs that aren't yet covered by an enabled recipient_rules row, scores them by frequency × total paise, and infers a likely reclassify_as target from (a) a small middleman keyword table and (b) whether the VPA shape matches any registered own-handle. No model involved. When the user accepts a suggestion we write a recipient_rules row with source = LEARNED (column already reserved in plans/schema.md:209).

State A — Settings → Recipient rules, with 3 suggestions

Suggestions live at the top of the existing recipient-rules screen under a Suggested (N) section, above the list of active rules. A confident suggestion (left border in accent, green Add rule button) is one where heuristics picked a concrete reclassify_as. An unsure suggestion (info-blue border, outlined button that reads Pick type…) is a repeat VPA where we don't have a default — user must choose what to reclassify as before accepting.

Resolved in v14

  • Storage. New rule_suggestions table (merchant, inferred_target, first_seen_ms, dismissed_at_ms). Survives restarts; dismissals stick across rescans.
  • Miner cadence.
    • Nightly WorkManager job — steady-state refresh.
    • One-shot kick on historical-import completion — fresh installs get suggestions immediately.
    • Rescan affordance on the Suggested header — user can force a re-mine (e.g. after adding an account, which changes own-handle inference).
    • Live-SMS BroadcastReceiver does not trigger the miner — a single row rarely crosses N.
  • Threshold. N ≥ 3 over 60 days to start. User-tunable later if too chatty / too quiet.

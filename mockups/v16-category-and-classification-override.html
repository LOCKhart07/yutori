v16 — Classification + Category override

Issue #132. Today the classifier writes classification + category at ingest time and never lets the user touch them. This mockup adds two override layers, kept symmetrical for both fields:

  • Per-transaction overrides on the tx-detail screen — one sheet per field, each with a "Use automatic" entry that clears the override and re-runs inference.
  • Rule-level assignments on the recipient-rule add/edit form — both fields are independently nullable. Reclassify as gets a new "Don't change" entry, so the rule can be category-only (e.g. tag swiggy-newbrand@paytm as Food & Dining without touching its UPI_PAYMENT classification).

The rule form and the per-tx sheet stay separate (Edit classification, Edit category — two buttons, two sheets). We may later merge them into one combined sheet; out of scope for now.

State A — Rule form: category-only rule (the Swiggy case)

The new "Don't change" entry sits at the top of the Reclassify as menu, visually distinct (info-tinted) so it reads as "leave the parser's classification alone." Picking it keeps the rule active but classification-neutral; combined with an Assigned category, this is the category-only rule path.

State D — Edit classification sheet

Mirrors the existing Edit category sheet from this PR: scrollable list, "Use automatic" at the top, current-mode hint under the title, Save disabled until the selection actually changes state. Picking a non-automatic value flips classification_override = 1 and writes the user's choice into transactions.classification (and recomputes budget_effect via BudgetEffectMapper). Picking "Use automatic" clears the flag and recomputes classification + budget_effect by re-running the classifier path.

What this implies for schema + classifier (engineering notes)

Schema (v5 migration, amended in place since v5 isn't merged yet):
  • recipient_rules.reclassify_as — relax to nullable (table-recreate; SQLite won't drop NOT NULL with ALTER).
  • recipient_rules.assigned_category TEXT — already in PR.
  • transactions.category_override INTEGER NOT NULL DEFAULT 0 — already in PR.
  • transactions.classification_override INTEGER NOT NULL DEFAULT 0 — new.
  • transactions.classification_inferred TEXT + transactions.category_inferred TEXT — new. Snapshot of what the automatic path produced at write time. Lets "Use automatic" restore without re-parsing the SMS, and preserves the inference signal even if the user keeps overriding.
Classifier: add resolveClassification(raw, matchedRule) mirroring resolveCategory. If matchedRule?.reclassifyAs == null, return raw — this is the category-only path. budget_effect always derives from the final classification via BudgetEffectMapper.
Backup: rule export already round-trips assignedCategory; extend so reclassifyAs is omitted (not "SELF_TRANSFER") when the rule has none. Per-tx data isn't in the settings backup today — staying out of scope; tx export is ui-spec §11.
Suggestion miner: the existing branch already excludes category_override = 1; extend to also exclude classification_override = 1 so the miner doesn't keep nagging about merchants the user has explicitly handled by hand.

Notes on this frame

Logo is the existing ic_launcher_round.png — rendering directly from the repo asset so the design decisions (brush-stroke ゆ, muted teal, cream cutout, no gradients) are preserved exactly in-app. Same asset the launcher uses, so the visual identity between home-screen icon and About is immediate.

Wordmark is Inter 28sp semibold — not a separate brand asset; leans on the system font so the About surface doesn't require shipping a custom webfont.

Tagline wraps naturally on two lines at 240px; on very narrow devices it'll wrap differently but the rhetorical beat ("Not less. But with confidence.") survives.

Principle cards are numbered so they read as a list; each one's heading is the principle name, body is the supporting copy with emphasis on the load-bearing phrases (em class is on-brand muted-white, not italic, to avoid visual noise).

"Build" block reuses the Settings card style from AppUpdatesSection — same elevated-surface bg, same row heights, same divider. "Check for updates" is actionable and triggers the same flow as Settings → App updates → Check now (share the ViewModel).

Licenses link goes to Frame 2 (sub-screen). GitHub link fires Intent.ACTION_VIEW on https://github.com/LOCKhart07/yutori.

Notes on this frame

Android-standard pattern — a plain list of library names + license type, each row tap-able to open a third-screen with the full license text (not drawn here; same layout as any long-form Text surface).

Not using com.google.android.gms:play-services-oss-licenses — that's Play Services, tied to Play Store distribution and pulls in a large transitive dep. Yutori's no-Play-Store stance means we generate the list at build time from Gradle ourselves. The com.jaredsburrows.license or com.github.jk1.dependency-license-report plugins are the two mainstream picks; either emits a JSON asset the Compose screen reads at runtime.

The "generated at build time" footer is an honest declaration — if we go auto-generated, the list stays fresh across dependency bumps without someone remembering to edit this list.

Scope for this PR: the screen lands here; the auto-generation plugin decision can be a separate follow-up. Ship with a hand-curated list for v0.7.0, swap to generated output later.

## Why "Yutori" **Yutori** (余裕) is Japanese for "breathing room": financial margin, mental ease, room to spend without friction. The logo is the hiragana **ゆ** ("yu") rendered as negative space; the space itself is the design. ## Principles Yutori is an opinionated app. A few load-bearing principles decide what ships and what doesn't. **1. SMS-first, human-assist second.** Bank SMS is the source of truth. ATM withdrawals count as spend at withdrawal time. Manual entry stays light. No OCR, no email scraping. **2. Margin, not micromanagement.** The central question is *"how much room do I have right now?"* Yutori doesn't moralise about spend. It surfaces impact: threshold crossings and single-transaction alerts. Awareness, not judgment. **3. On-device by default.** You own your data. Yutori has no backend, no accounts, no analytics, no data collection. **4. Opinionated about scope.** One question, answered well: personal monthly spend visibility from bank SMS. Features that don't serve "how much room do I have" are out. **5. Side-loaded, open source.** Distributed via GitHub releases. No Play Store (fees don't fit a no-cost app). Updates land in-app via the built-in autoupdater. Code is public: audit it, fork it. > **You don't want to spend less. You want to spend confidently.**

The blockquote at the bottom is the tagline — set apart so it reads as the closing line, not just another sentence.

v19 — show real classification on the Message log (#152)

The Message log screen (shipped in #146) currently shows one of four rollup pills: Affects budget / Tracked as income / Ignored / Needs review. The user reports that the labels are opaque — “Needs review? Wtf is that” — and sensibly asks why the screen needs a translation layer at all. This mockup swaps the rollups for the actual Classification enum value, rendered via one canonical displayName. Color tone is retained so the quick visual scan (green = spend, blue = income, yellow = parser failed, grey = dropped) still works.

B — Canonical mapping

Today two private helpers do this job and disagree: prettyClassification in TransactionDetailScreen.kt:1013 (acronym-preserving tokenizer, produces “Non financial”, “Self transfer”) and classificationLabel in AddEditRecipientRuleScreen.kt:823 (hand-mapped, produces “Non-financial (drop)”, “Self-transfer”). One canonical Classification.displayName next to the enum in :parser, used by all three screens (transaction detail, rule picker, message log). Table below is the proposed canonical form.

EnumdisplayNameTone (A2)Notes
CC_TRANSACTION CC transaction spend
UPI_PAYMENT UPI payment spend
DEBIT_CARD Debit card spend
ATM_WITHDRAWAL ATM withdrawal spend
REFUND Refund spend Negative SPEND, same tone.
CASHBACK Cashback spend Refund-effect today.
INCOMING_CREDIT Incoming credit income
CC_BILL_PAYMENT CC bill payment drop Current rule-picker label was “CC bill payment” — keep as-is.
SELF_TRANSFER Self-transfer drop Hyphenated, matches rule-picker label.
NON_FINANCIAL Non-financial drop Drop the (drop) parenthetical — context-specific to the rule picker.
OTP OTP drop All-caps acronym, no rewrite.
BALANCE_ALERT Balance alert drop
UNMATCHED Unmatched review The one tone that isn’t driven by BudgetEffect — UNMATCHED has no budget effect, so it gets its own warn tone.
Sequencing. Two commits. (1) refactor: promote Classification.displayName to :parser, delete both private helpers, route transaction detail & rule picker through it. Pure rename; no visual change on those two screens (“Non-financial (drop)” on the rule picker becomes “Non-financial” — tiny copy change, arguably cleaner). (2) feat: swap the Message log pills to use it + drop the rollup mapping. Issue #152 closes on (2).
IngestedMessageOutcome enum goes away. LatestIngestedMessage.outcome: IngestedMessageOutcome becomes LatestIngestedMessage.classification: Classification? (nullable because old rows may have UNMATCHED or a string that no longer decodes — today that’s folded into NEEDS_REVIEW; after, it’s rendered as Unmatched the same way). The rollup/outcome type had exactly one caller; deleting it is cleaner than keeping it as a second taxonomy.
Rule-picker behaviour is unchanged. AddEditRecipientRuleScreen still filters its picker to the reclassify-addressable subset (CC_BILL_PAYMENT, SELF_TRANSFER, REFUND, INCOMING_CREDIT, NON_FINANCIAL). Only the label strings come from the canonical helper — the filtering logic is untouched.
Not in scope. No filter chips on the log screen, no tap-to-reclassify, no search. Those remain “if the user then asks” follow-ups, same as #146.

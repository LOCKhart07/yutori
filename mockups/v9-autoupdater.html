v9 · In-app autoupdater — update dialog + settings row (#71)

Two surfaces gated on mockup approval per plans/autoupdater-spec.md §7: the update-available modal that appears on cold-start when a newer release is detected, and the Settings → App updates section that houses the toggle + manual Check now. A third copy tweak on the restricted-settings first-run screen is a one-liner and doesn't need its own mockup — noted at the bottom.

Decisions locked (2026-04-17): dialog = AlertDialog, with the long-notes mitigations in section D below (pinned actions + expanded max-height). Status wording = "Update available: 0.3.0 ›". Check now = dedicated button. Error CTA = "Try again".
Sections A & B are kept as the design record — D is the chosen variant for slice 5.

A · Modal update-available sheet (bottom sheet)

Cold-start trigger fires on the dashboard. Sheet slides up; dashboard stays visible underneath (dimmed) so the user has context for what they're leaving. Scrolls independently if release notes are long. Swipe-down / Later dismisses and sets dismissed_tag = v0.3.0 so the next cold-start within 6h stays silent.

B · AlertDialog alternative (centre-stage)

Standard Material AlertDialog — cheaper to wire (one Compose primitive, no ModalBottomSheet state) and visually consistent with Android's own system confirmation UI that fires right after "Download & install" anyway. Trade-off: cramped for long release notes; the markdown area scrolls but it's a smaller window than a sheet.

D · AlertDialog — final (pinned actions + expanded height)

What slice 5 builds. AlertDialog + DialogProperties(usePlatformDefaultWidth = false) for a wider-than-default panel that fills ~82% of the screen. The release-notes body scrolls independently within the panel, while Later / Update stay pinned at the bottom — no matter how long the notes are the user can always act. Fade gradient at the bottom of the scroll area hints "more below" without a scrollbar. Optional "See full notes on GitHub" escape hatch at the bottom — pairs naturally with #77 once that lands.

Decisions (locked 2026-04-17)

1. Dialog shape — D (AlertDialog with pinned actions + expanded height). A & B kept as design record.
2. Status-line wording — "Update available: 0.3.0 ›" (per C2).
3. "Check now" placement — dedicated button below the card (per C1–C5).
4. Error-state CTA — "Try again" in both the dialog (D3) and the settings row (C4).

Third surface, not mocked: spec §7.3 adds a single line to the existing restricted-settings first-run screen (v6-restricted-settings.html): "Once Yutori is installed, future updates arrive as an in-app prompt and install without another Play Protect warning." — inline copy only, no layout change.
